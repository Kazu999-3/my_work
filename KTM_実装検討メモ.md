## 1. プレイヤーページ

### P-04 代表MMR表示ロジックの統一
- **目的**: profile履歴の TOTAL がまだ単純平均で、N1(重み付け)と不一致。表示を一貫させる。
- **方針**: `api/player/profile/route.ts` の `matchMmr.TOTAL`（現状 `(top+..+sup)/5`）を、共通関数 `computeRepresentativeMmr`（`lib/mmr.ts`）で算出。ただし履歴の各時点では「その時点までのレーン別試合数」が必要なので、逆算ループ内でレーン別ゲーム数もカウントしながら重み付けする。
- **データ変更**: なし（計算のみ）。
- **注意**: 逆算（新→古）でゲーム数を減算する順序に注意。厳密一致が難しければ「表示は現在の games_* で近似」でも可。
- **規模**: 小〜中。

### P-05 調子サマリーの深掘り
- **目的**: 連勝連敗に加え「レーン別の最近の調子」「時間帯別勝率」を出す。
- **方針**: `player/[id]/page.tsx` の `recentForm` memo を拡張。history から role別に直近N戦の勝率、`created_at` の時間帯(JST hour)別勝率を集計。UIは既存の調子カードにタブ/小見出し追加。
- **データ変更**: なし（history に role/date あり。時間帯は created_at から算出）。
- **注意**: サンプル数が少ないレーン/時間帯は「データ不足」表示。
- **規模**: 中。

### P-06 チャンピオンプール可視化
- **目的**: 使用チャンプの幅・勝率・熟練度を一目で。
- **方針**: profile が返す `stats[role].champions`（name/games/wins/winRate）と `main_champions`(マスタリー) を利用。UIは新規タブ or summary内にバブル/ヒートマップ（recharts散布 or 単純なグリッド）。X=試合数, Y=勝率, サイズ=マスタリーpts。
- **データ変更**: なし。
- **注意**: recharts依存は既存。チャンプ数が多い時の描画量。
- **規模**: 中。

### P-07 ライバル/宿敵カードの常設
- **目的**: よくキル/されする相手をプロフィール上部に常設。
- **方針**: 既存の chemistry/rivals API（`api/player/chemistry`）が nemesis/prey を返しているか確認し、profile 上部にカード追加。無ければ ktm_match_participants から kill関係を集計する軽いAPIを追加。
- **データ変更**: なし（既存集計を利用）。
- **注意**: キル情報の粒度（KDAはあるがキル相手の個別記録があるか要確認。無ければ「対面勝敗ベースの天敵」に代替）。
- **規模**: 小〜中（データ有無で変動）。

### P-10 モバイルのタブ操作改善
- **目的**: プロフィールのタブ（summary/lanes/chemistry/champions/history）がスマホで窮屈。
- **方針**: タブバーを横スクロール可＋現在地インジケータ、またはアイコン＋短縮ラベル。`player/[id]/page.tsx` のタブ描画部のみ。
- **データ変更**: なし。
- **規模**: 小。

## 3. チーム分け（balancer）

### BL-01 探索のWeb Worker化
- **目的**: 数秒UIが固まる問題の解消（挙動不変）。
- **方針**: `lib/balancer.ts` の `runBalanceSearch`/`coreBalanceProposals` を Web Worker で実行。`balancer/page.tsx` から Worker に players/ctx を postMessage → 結果受信。純関数なので移植は容易。Next.jsのWorkerは `new Worker(new URL('./balanceWorker.ts', import.meta.url))`。
- **データ変更**: なし。
- **注意**: SSR回避（クライアントのみ）。型の共有。フォールバック（Worker不可環境）は同期実行。
- **規模**: 中。

### BL-02 上位候補数の調整UI
- **目的**: 精度/速度のトレードオフを管理者が選べる。
- **方針**: `runBalanceSearch` の `topCandidates = screenResults.slice(0, 100)` の 100 を引数化。balancer管理パネルに「探索強度: 速い/標準/精密(40/100/200)」セレクト。
- **データ変更**: なし。
- **規模**: 小。

### BL-03 禁止ペアをdiscord_id基準に
- **目的**: 名前一致の脆さ(同名/表記ゆれ)を解消。
- **方針**: 禁止ペア設定を「名前ペア」→「discord_idペア」に。設定元（ktm_settings等）のスキーマとUIを id ベースに変更。`balancer.ts` の `checkSameTeam` は Player.discordId で比較（Player型に discordId は既にある）。移行: 既存の名前ペアを id に変換するワンショット。
- **データ変更**: 設定の保存形式変更（migration or 設定JSON更新）。
- **注意**: discord_id 未登録プレイヤーの扱い（名前フォールバック併用）。
- **規模**: 中。

### BL-04 固定ロールスロットの並べ替え除外
- **目的**: perm削減でさらに高速化。
- **方針**: `runBalanceSearch` の perms 生成を「固定ロールのスロットを除いた自由スロットのみ順列」に変更。固定枠は先に確定し、残りだけ `getPermutations`。挙動は等価（現状は全perm生成後に validA/validB で弾いている）。
- **データ変更**: なし。
- **注意**: インデックス対応が複雑になるので慎重に（テスト必須）。BL-01と併せると効果大。
- **規模**: 中。

### BL-05 チーム分け結果の満足度学習
- **目的**: 満足度投票(#42)を係数調整にフィードバック。
- **方針**: 満足度データ（既存の投票テーブル）を集計し、「満足度が低かった構成の特徴（大きいレーン格差/オフロール多）」を分析。まずは**可視化**（管理パネルに満足度と各ペナルティ指標の相関表）→ 手動で係数調整、を第一段階に。自動学習は第二段階（重み探索）。
- **データ変更**: なし（既存投票 + 予測テーブル）。
- **注意**: サンプル数が少ないと過学習。まずは可視化＋手動が現実的。
- **規模**: 大（自動化まで行くなら）。段階実装推奨。

### BL-06 「なぜこの分け方か」の説明強化
- **目的**: 各人の配属理由をより具体的に。
- **方針**: `buildBalanceResult` の `balanceReport` を拡張。各サブロール配属者に「本来◯希望/競合◯人/格差◯」等の数値根拠を付与。既にロジック内に penalty 要素があるので、その内訳を理由文へ。
- **データ変更**: なし。
- **規模**: 中。

### BL-08 観戦ローテの公平性可視化
- **目的**: 誰が何回観戦したかの履歴。
- **方針**: spectator_pity（既存）や観戦記録から、直近の観戦回数を集計して名簿/バランサーに表示。データ源が無ければ recruitments/participation から算出。
- **データ変更**: 記録が無ければ観戦ログ列/テーブル追加の検討。
- **規模**: 小〜中。

- **目的**: 「勝敗+18 / Elo+5 / KDA+3 / ダンパー×0.9」等の内訳を見せる。
- **方針**: `lib/mmr.ts` の `calculateNewMMR` を、数値だけでなく内訳オブジェクトも返す版（`calculateNewMMRDetailed`）を追加 or 既存を拡張。試合記録(`api/match/record`)時に内訳を `ktm_match_participants` に保存、プレイヤーページのツールチップ/履歴で表示。

- **注意**: 再計算だと当時のcurrent/opponent MMRが必要（履歴逆算で近似可）。保存が確実。

- **方針**: `api/mmr/check-integrity` のロジックを GitHub Actions cron（既存のバッチ基盤）で定期実行し、ズレ>閾値なら Discord webhook / Web Push で管理者通知。

- **規模**: 中。

### M-06 代表MMRの重み付けを表示にも反映
- **目的**: P-04と同一。leaderboard等の表示 mmr が weighted で一貫しているか点検。
- **方針**: 表示は `ktm_players.mmr`（既に weighted 保存）を使うだけ。二重計算している箇所が無いか棚卸し。P-04とセットで対応。

- **注意**: 認証基盤の変更なので影響大。段階導入（まず2段階: 閲覧/管理）。
- **規模**: 大。

### A-05 ダッシュボードのヘルス詳細化
- **目的**: DB/API/Riot/最終同期時刻を個別表示。
- **方針**: `api/health` を拡張（DB接続, Riot APIキー有無/疎通, 最終同期時刻, Geminiキー有無）。`SystemStatus` or 管理ダッシュボードに個別インジケータ。
- **データ変更**: なし。
- **注意**: Riot疎通チェックはレート消費。軽量に（キー存在確認＋最終成功時刻）。
- **規模**: 小〜中。

## 11. セキュリティ / 運用
