# Auto-Deploy & Auditor (自律型デプロイ＆監査プラグイン)

このワークフローは、ユーザー（マスター）からの「デプロイして」という明示的な指示、またはAI自身が重要なコード改修を完了した際に呼び出される自動リリースプロセスです。
AIは以下の手順に従い、**「ローカルでの検証 ➔ デプロイ ➔ 本番環境の自己監査 ➔ 通知」** を完全に自律して実行しなければなりません。

## 🛡️ Step 1: ローカル・プレビルド監査 (Pre-build Check)
コードを本番へPushする前に、必ずローカルでビルドが通るか確認します。
1. フロントエンド（`99_ARCHIVE/04_COMMAND_CENTER_old`）の変更がある場合：
   - ターミナルで `cd 99_ARCHIVE/04_COMMAND_CENTER_old` を実行。
   - `npm run build` を実行し、JSの構文エラーや依存関係エラー（Fatal error）が出ないことを確認する。
   - エラーが出た場合は、**デプロイを即座に中止し、自律的にコードを修復して再ビルド**する。

## 🚀 Step 2: デプロイの実行 (Deployment)
ビルドが正常に通った場合のみ、本番環境への変更の適用（Git Push）を行います。
1. `git add -A`
2. `git commit -m "chore: auto-deploy by auditor"` (適切なメッセージを付与)
3. `git push origin master` (これによりVercel等のフックが走る)

## 🩺 Step 3: セルフ監査 (Health Audit)
これが最も重要なステップです。デプロイが完了したとみなされる数秒〜数十秒後、**本番URLが正常に動いているかAI自身がチェック**します。
1. 本番のURL（Vercelのドメイン等）に対して、`run_command` を用いてアクセスチェックを行う。
2. （必要であればPlaywright等でヘッドレスブラウザを開き、「画面が真っ暗（White screen of death）」になっていないか、JSのコンソールエラーが出ていないかを検証する）。
3. 異常（500エラーや画面描画の失敗）を検知した場合は、ユーザーに報告する前に**「自動ロールバック（git revert）」または「自己修復処理」**を行うこと（SREフェーズへ移行）。

## 📢 Step 4: 完了通知 (Notification)
監査が完了し、本番環境が正常に稼働していることを確認できたら、ユーザーへ通知を行います。
この通知を作成する際は、必ず **「Notification Designer (通知デザイナー) スキル」** の絶対ルールに従ってください。
- 絵文字を使うこと。
- 本番URLの直リンクを付与すること。
- （例: `✅ **本番環境へのデプロイと監査が完了しました！** [👉 Webポータルを開く](https://... )`）
