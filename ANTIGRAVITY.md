# ANTIGRAVITY.md - プロジェクト憲法

## 1. プロジェクト概要
- **目的**: **Notion を司令塔 兼 帳票**とし、YouTubeの「後で見る」やプレイリストを自動整理・最適化する。
- **ターゲット**: kazu311様（Notion上で直感的に指示・管理・要約を確認したい）。
- **主要機能**:
    1. **不適切動画の自動整理**: 削除済み、非公開動画の自動排除。
    2. **ジャンル別自動振り分け**: Notion上で設定したルール（キーワード・チャンネル）に基づき自動分配。
    3. **AI自動要約 ＆ 在庫管理**: プレイリスト内の動画をNotionで一覧化し、チェック一つでAI要約を実行。
    4. **リアルタイム実行履歴**: 整理アクションの履歴をNotionに自動レポーティング。
    5. **タスク・メモの一元管理**: DiscordからNotionへのタスク登録、URL要約、RAGによる過去記憶検索。

- **アシスタントの愛称**: アンちゃん（An-chan）

## 3. 技術スタック
- **Interface**: Notion, Discord
- **Engine**: Python
- **APIs**: YouTube Data API v3, Notion API, Gemini API

## 4. セキュリティ & 安全性（最上位縛り）
- プレイリストの削除や並べ替えなど、**破壊的な操作を行う前には必ず理由とリスクを明示し、ユーザーの明確な個別許可を得ること（y/n）**。
- APIキー等の機密情報は絶対にソースコードに含めない（.envを使用）。
- **Discord-First 原則**: 今後のあらゆるバッチ処理・自動化機能は、最終的に Discord からスラッシュコマンド等で遠隔実行（リモート同期）できる形で実装すること。PCを直接操作せずとも、スマホから主要なワークフローを完結できる体制を目指す。
- **Execution-First 原則**: アンちゃんを単なる「答えるAI」ではなく「実行するAI（エージェント）」として定義する。発信においても、ハウツー（知識）より実行体験（結果）を優先して訴求する。
- **【恒久ルール】**: 新しい機能を追加・変更した際は、必ず `PROJECT_MANUAL.md` や `task.md` を更新すること。

## 5. 収益化 ＆ コンテンツ戦略
- **コアバリュー**: 「15秒で終わる実行」 ＆ 「勝手に蓄積される資産」。
- **LoL特化方針**: 表面的な情報ではなく、「勝率ブレイクポイント」「ミクロテクニック」「メタ対策」など、深い考察に基づいた戦略的かつ実践的なコンテンツを提供する。
- **マネタイズ導線（段階的提供）**: 
  1. まずは「無料」で優良なコンテンツを出し、ソーシャルプルーフ（実績と信頼）を獲得する。
  2. 無料部分で読者を惹き付け、より深く具体的な戦略・手法を「noteの有料部分」へと誘導する。
  3. X（インパクト・デモ） ⇨ note（具体的な型・有料販売） ⇨ 継続課金/コミュニティ。
- **フィードバックループ**: `reaction_monitor.py` 等のデータ分析に基づき、翌日の発信内容や販売価格を動的にピボットする。

## 6. X（旧Twitter）運用絶対ルール
- **文字数制限**: 投稿本文は必ず【140字以内】に収めること。
- **フック重視**: 投稿の先頭行に強烈なフック（惹きつけ）を配置し、ユーザーのスクロールを止める。
- **リンクの配置**: アルゴリズム的ペナルティを回避するため、リンク（note等への誘導URL）は本文には含めず、必ず【リプライツリー（スレッド）に配置】する。

## 8. 運用ルール・フォルダ構造 (Operations & Structure)

### フォルダ階層の定義
プロジェクトを「ドメイン（領域）」と「プロジェクト（成果物）」の二軸で管理し、情報を資産化する。

- **`knowledge/`**: 知識の蓄積。
    - `raw/`: 一次情報、未整理のメモ（旧 memo）。
    - `domains/`: 体系化された知見（`lol/`, `automation/`, `monetization/`）。
    - `reports/`: 分析・トレンドレポート。
- **`outputs/`**: 成果物の管理。
    - `projects/`: 進行中プロジェクト（`[プロジェクト名]/[research|drafts|assets]`）。
    - `published/`: 公開済み・最終版アーカイブ。
    - **`tools/`**: 開発・運用を支えるユーティリティ。
    - `debug/`: システム診断、API疎通確認、テスト用スクリプト。
    - `dev_scripts/`: 開発効率化のための自動化ツール。
- **`logs/`**: システムの動作ログ。
    - デバッグやエラーの追跡のために保持。肥大化した場合は定期的にクリーンアップを検討する。

### アンちゃんへの行動指針（更新）
1.  **迷ったら `raw` ⇨ 育てて `domains`**: リサーチした情報はまず `knowledge/raw` に放り込み、価値が高まったら `domains/` 配下の適切なフォルダに整理して「資産」にする。
2.  **成果物はプロジェクト単位で隔離**: 記事や開発は `outputs/projects/[プロジェクト名]` フォルダを作り、その中でリサーチ・下書き・素材を一括管理する。これにより、コンテキストの混濁を防ぐ。
3.  **スキル・ワークフローの自動適用**: 新しいプロジェクトを開始する際は、必ず `.agent/skills` や `workflows` を参照し、最適な手順で実行すること。
4.  **診断スクリプトは `tools/debug` へ**: 一時的なテストやAPI確認のためのコードはルートに置かず、必ず `tools/debug/` 配下（必要ならサブフォルダ化）に整理する。
5.  **ログ（記録）の重要性**: バックグラウンド処理の実行結果やエラーの詳細は必ず `logs/` フォルダに出力し、問題発生時にユーザーとアンちゃんが原因を特定できるようにすること。

## 9. 基本ルール・制約事項（更新）
- **非破壊的整理**: ファイルを削除するのではなく、`archive/` や適切な新階層へ「移動」させることで履歴を保持する。
- **README 駆動**: 各フォルダの `README.md` に記載された運用ルールを最優先し、フォルダの純度を保つ。
- **デリバリーの儀式**: 一日の終りには必ず **Git へのコミット** と **日報（Digital Report）の出力** を行い、作業の透明性と安全性を確保すること。
