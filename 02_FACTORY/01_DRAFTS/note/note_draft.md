# 動画を観ずに1秒で記事化！YouTube自動化AI

ネット上の情報収集で、誰もが一度は直面する壁があります。それが「動画リサーチにかかる膨大な時間」です。

有益なノウハウや攻略情報がYouTubeに数多く投稿されている一方で、それらを1本ずつ数十分かけて視聴し、一時停止を繰り返しながらメモを取る作業は、あまりにも非効率です。英語の解説動画であれば、翻訳の手間も重なり、1本の要点をまとめるだけで数時間が吹き飛ぶことも珍しくありません。

この時間の浪費を解決し、URLを登録するだけで、AIが動画の全音声を解析して完璧な要点まとめ記事（日本語）を裏側で自動量産するシステム「YouTube Absorber」を構築しました。本記事では、このシステムの具体的な仕組み、技術的な実装フロー、そして開発過程で直面した生々しい失敗談と解決策のすべてを公開します。

## 情報収集の生産性を10倍にする自動化システム

動画を観るという行為は、極めて時間効率が悪いインプット方法です。テキストであれば数秒で流し読みできる情報量であっても、動画では10分や20分拘束されます。この課題をシステム的に解決するのが、今回紹介する「YouTube Absorber」です。

システム全体の構成は以下の図のようになっています。ポータル（Next.jsで構築した管理画面）に動画のURLを登録すると、裏側で動く自律型のPythonプログラムとAI（Gemini API）が連動し、最終的に整理された攻略データベースへ自動的にマージ（統合）されていくイベント駆動型の構造です。

```mermaid
graph TD
    Admin[管理者] --> |URL登録| Portal[Web Portal (Next.js)]
    Portal --> |キュー登録| Queue[youtube_queue (Supabase)]
    SRE[SRE Daemon] --> |15分ごとに自動起動| YTA[YouTube Absorber]
    YTA --> |音声ストリーム取得| yt[yt-dlp]
    YTA --> |ローカルGPU推論| Whisper[faster-whisper large-v3]
    Whisper --> |超高精度テキスト| Gemini[Gemini API]
    Gemini --> |Markdown記事返却| YTA
    YTA --> |DB保存| DB[(Supabase DB)]
```

この自動化フローの最大の特徴は、単なる「動画の書き起こし」で終わらない点にあります。動画の内容を構造化されたMarkdown記事に仕立て上げ、それを同一ジャンル（例えば、League of Legendsの特定チャンピオンや汎用的なマクロ戦術）の既存の「総合教科書」にAIが自動で追記マージしていく仕組みまで組み込まれています。

さらに今回、YouTube側の「自動字幕（英語）」に依存していた従来のやり方を廃止し、**「常にローカルのGPUパワーをフル動員して最高精度の音声認識AIで文字起こしを行うクオリティ特化型」**へとシステムを進化させました。

## YouTube動画を記事に変換する4つのステップ

このシステムが、1本の動画URLからどのように記事を自動量産しているのか、裏側で動いている4つのプロセスを解説します。

### ステップ1：音声ストリームの高速ダウンロード
まず、管理画面から登録された動画URLを受け取ると、Pythonの裏側で動画情報取得ツールである `yt-dlp` が起動します。
動画そのものをダウンロードするとディスク容量やダウンロード帯域を大量に消費するため、システムは映像を除外した「最高音質オーディオストリーム（.webm や .m4a などの生の音声ファイル）」のみをピンポイントで取得します。この際、再変換（エンコード）のポストプロセスをあえて通さないことで、ffmpegなどの外部エンコーダを必要とせず、わずか2〜3秒で音声ダウンロードを完了させます。

### ステップ2：ローカルGPUによる超高精度書き起こし（Whisper large-v3）
取得した生の音声ファイルは、ユーザーのローカルPCに搭載されている **NVIDIA GeForce RTX 3060 Ti (VRAM 8GB)** を使用し、ローカルで音声認識AI `faster-whisper` を走らせてテキスト化します。
使用するAIモデルは、OpenAIがリリースした最高精度の **`large-v3`（float16精度）** です。VRAM消費量は約3GBに抑えられているため、VRAM 8GBのグラフィックボードがあれば、他のPC作業を全く邪魔することなく、超高速（45分の解説動画をわずか約5.9分）で極めて高精度な書き起こしが完了します。
YouTubeの自動字幕と異なり、文脈からピリオドやカンマを正確に補完し、ゲーム用語（「Shyvana」「Q start」「Nafiri」など）の聞き間違いを完璧に防止します。

### ステップ3：AIによるコンテキストに応じた構造化・日本語化
書き起こされた高精度なテキストを、Gemini API に流し込みます。
この際、単に「翻訳して要約してください」と指示するだけでは、ゲームの専門用語やスラングが直訳されてしまい、支離滅裂な文章が出力されます。そこでプロンプトに「LoL（League of Legends）の専門知識を持つチャレンジャーコーチ」という明確な役割を与え、以下のようなテンプレートに従って情報を再構成するように指示します。
- **獲得した教訓・動画の結論**（1行サマリー）
- **マクロ戦略・ルート・判断基準**（なぜその選択をしたかの理由付け）
- **ミクロ・戦闘のコツ**（スキルコンボ、ガンクのタイミング等）
- **重要な金言（名言・Tips）**
これにより、単なる直訳ではない、プレイヤーにとって真に価値のある日本語の「攻略バイブル」に変換されます。

### ステップ4：総合バイブルへの自動マージ（Dict Synthesizer）
バラバラに登録された個別の動画まとめ記事を放置しておくと、情報が散乱して読みづらくなります。そこで、定期的にバックグラウンドで `Dict Synthesizer` というプログラムが起動します。
このプログラムは、同じキーワード（例：「マクロ」「ジャングル」）を持つ個別の記事をロードし、Gemini API を使って既存の「総合マクロ教科書（総合バイブル）」へ新しい知見を矛盾なく追記マージします。
マージが完了すると、元の個別の動画まとめ記事のキーワードに `__DELETED__` というタグを付与し、お役御免となった個別ファイルをクリーンアップデーモンが物理的に自動削除します。

## 開発中に直面した「4つの致命的な罠」と解決策

このシステムを24時間安定して稼働させるまでには、いくつかの手痛い失敗とバグに直面しました。これらをどうやって修正したか、その生々しい解決策を共有します。

### 1. Windows + Python 3.14.3 環境における CUDA DLL ロードの壁
ローカルの GPU で Whisper を動かそうとした際、最大の障壁となったのが Python のバージョンと Windows の仕様変更でした。
現在使用している Python 3.14.3 は新しすぎるため、PyTorch の公式 CUDA WHL パッケージが存在しませんでした。また、Windows上の Python 3.8 以降ではセキュリティ対策で DLL 検索ポリシーが変更されており、`nvidia-cublas-cu12` などのパッケージを pip でインストールしても、`ctranslate2`（Whisperの高速推論エンジン）が `cublas64_12.dll is not found` とエラーを吐き、GPUでの初期化に失敗していました。

*   **解決策**:
    PyTorch に依存せず、直接 C++ の CUDA ライブラリを読み込むため、`nvidia-cublas-cu12` と `nvidia-cudnn-cu12` を Python 仮想環境に導入。さらに、環境変数の追加だけでは解決しないロードエラーを回避するため、**「仮想環境内の nvidia パッケージから、`cublas64_12.dll` や `cudnn64_9.dll` などの DLL 群を、`ctranslate2` パッケージのディレクトリ直下へスクリプトで直接コピー・物理配置する」** という設計を適用しました。これにより、Python 3.14 環境であっても一切の環境バグを起こさず、100%確実に GPU (CUDA) での超高速推論を走らせることに成功しました。

### 2. ffmpegの未インストールによるダウンロード異常
`yt-dlp` で音声をダウンロードする際、最初は「m4a形式への抽出・再エンコード（ポストプロセス）」をかけていました。しかし、この処理はシステムに `ffmpeg` / `ffprobe` がインストールされていることを前提としているため、Windows環境でこれらのパスが通っていない場合に `Postprocessing: ffprobe and ffmpeg not found` というエラーで解析がストップしてしまいました。

*   **解決策**:
    音声をわざわざ別フォーマットに再変換（エンコード）するのをやめ、`-f ba` オプションで YouTube から「生の最高音質オーディオストリーム（webm形式など）」をそのまま無変換でダウンロードする設計に変更しました。Whisperのバックエンド（PyAVデコーダー）は、ファイルが webm や mp4 のままであっても直接デコードできるため、結果として **ffmpegのインストールを不要にし、かつエンコード処理を無くしたことでダウンロード速度を約5倍に引き上げる** という一石二鳥の解決をもたらしました。

### 3. 不要データによるデータベースの肥大化
動画を登録するたびに新しいレコードが追加され、それをマージした後の「古い元記事」が残り続けることで、データベースの容量が急速に圧迫される問題が発生しました。

*   **解決策**:
    自動マージが終わった時点で、マージ元の記事レコードのキーワード配列に `["__DELETED__"]` を上書きします。そして、15秒間隔で常駐して動き続けるクリーンアップ用のデーモンスレッドが、このフラグがついたレコードを検知し、サーバー上のローカルファイルを物理的に削除した上で、Supabaseのテーブルから完全にレコードを抹消（DELETE）する「自動お掃除機構」を実装しました。これにより、データベースは常に最小限のクリーンな状態を維持できるようになりました。

### 4. Windows環境でのファイルロック（PermissionError）
SRE Daemonがログファイル（`sovereign_os.log`）を読み込んで常時エラー検知を行っている際、Pythonプロセスがログファイルをオープンしたままにしてしまい、他のプロセスがログを書き込もうとしたときに `PermissionError` が多発し、システム全体が沈黙するバグが発生しました。

*   **解決策**:
    ファイルを「開きっぱなしにする」ループ処理を完全に廃止しました。ファイルを読み取るたびに必ず即座にクローズ（`with open(...)` の適切なスコープ管理）し、次に読み取る際には前回の最終読み取り位置（`tell()` で取得したバイト数）をメモリに保存しておき、`seek()` でその位置までスキップして差分だけを高速に読み取るロジックへリファクタリングしました。これにより、Windows特有のファイルロック競合を完璧に回避することに成功しました。

## 今すぐ自分の環境で情報収集を自動化する手順

この「動画からテキスト記事を自動生成する」仕組みは、個人ブログの執筆や Notion でのナレッジ蓄積にも広く応用可能です。今日から始められる具体的なアクションプランを示します。

1.  **コマンドラインツールの導入**
    パソコンに `yt-dlp` をインストールし、コマンドプロンプトやターミナルから以下のコマンドを実行してみてください。
    ```bash
    yt-dlp -f ba "動画のURL"
    ```
    これで、映像を除外した生のオーディオストリーム（webm または m4a ファイル）だけがローカルに一瞬で保存されます。
2.  **ローカルでの音声認識体験 (Whisper)**
    Python環境があれば、以下のコマンドで簡単に `faster-whisper` を使ってローカルで書き起こしが可能です。
    ```bash
    pip install faster-whisper
    ```
    簡単な Python スクリプトで `WhisperModel("large-v3", device="cpu", compute_type="int8")` を動かすだけで、クラウドAPIを叩くことなく無料で高精度なテキストが得られます（GPUがあれば `device="cuda"` でさらに高速化できます）。
3.  **AIによる要約とデータベース化**
    出力されたテキストをChatGPTやGeminiに流し込み、自分の欲しいフォーマット（ブログ記事用、勉強ノート用など）を指定して要約させます。これを Notion などに集約していくことで、自分だけの「AIナレッジライブラリ」が完成します。

動画をすべて最初から最後まで観る時間は、もう必要ありません。テクノロジーを賢く使って、情報収集の効率化へ一歩踏み出してみましょう。
